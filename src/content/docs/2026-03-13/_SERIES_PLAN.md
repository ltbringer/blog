# Blog Series Framework: Qdrant Internals for Contributors

## Context
A multi-part blog series targeting Rust developers / potential contributors, going all the way to algorithm depth. This plan provides **topic ordering, TOC structure per post, and source material pointers** — not content. You write the posts yourself.

## Series Structure (10 parts)

Top-down intro, then lifecycle-driven deep dives, then bottom-up algorithm posts, then distributed systems.

```
Part 1:  Bird's Eye Architecture (what are the crates, how they layer)
Part 2:  The Life of a Query (search request lifecycle, end-to-end)
Part 3:  The Life of a Write (upsert lifecycle, WAL, consistency)
Part 4:  Segments — The Storage Primitive (what's inside, mutable vs immutable)
Part 5:  The Life of an Index (plain → HNSW, optimizer triggers, proxy segments)
Part 6:  HNSW Deep Dive (the algorithm, construction, search, GPU)
Part 7:  Quantization & Scoring (scalar, PQ, binary, scoring pipeline)
Part 8:  Payload Indexing & Filtering (field types, cardinality estimation, query planning)
Part 9:  Sparse Vectors & Hybrid Search (inverted index, posting lists, fusion)
Part 10: The Distributed Life (bootstrap, consensus, replication, shard transfer, replica states)
```

## Part 5: The Life of an Index — Optimization & Segment Lifecycle

**Goal:** How segments evolve: plain → indexed, small → merged, deleted → vacuumed. The proxy segment pattern. Resource budgeting.

### TOC
1. The optimization loop: `UpdateHandler::optimization_worker_fn()` listens for signals
2. Four optimizers and when each fires:
   - **Indexing**: segment grows past `indexing_threshold_kb` → build HNSW
   - **Merge**: too many small segments → merge the 3 smallest
   - **Vacuum**: deletion ratio > `deleted_threshold` (e.g. 20%) → compact
   - **Config mismatch**: HNSW params, quantization, or storage type changed → rebuild
3. `OptimizerThresholds`: the three knobs (`max_segment_size_kb`, `memmap_threshold_kb`, `indexing_threshold_kb`)
4. The Proxy Segment pattern (copy-on-write during optimization):
   - Original replaced with `ProxySegment` wrapper
   - All writes go to proxy (tracks deletes, index changes)
   - Original stays read-only for in-flight queries
   - New optimized segment built in background
   - Final swap: apply recorded changes, replace proxy → optimized segment
5. Resource budgeting: CPU/IO permits, concurrency limits
6. What happens on cancellation? (rollback via `unwrap_proxy()`)
7. `TrackerLog`: optimizer execution history and monitoring

## Part 6: HNSW Deep Dive

**Goal:** The algorithm that makes Qdrant a *vector* database. Construction, search, parameters, GPU acceleration.

### TOC
1. Why HNSW? (approximate NN search, O(log n) vs brute force)
2. The graph: hierarchical layers, navigable small world property
3. Graph construction: inserting a new point (layer selection, neighbor selection, bidirectional links)
4. Search: greedy traversal from top layer down, `ef` parameter controls beam width
5. Configuration and tradeoffs:
   - `m` / `m0` — connections per node (accuracy vs memory)
   - `ef_construct` — build-time beam width (build speed vs quality)
   - `ef` — query-time beam width (latency vs recall)
   - `full_scan_threshold` — brute force cutoff
6. Filtered search: `FilteredScorer` integrates payload filters with graph traversal
7. Link storage: compressed serialization format
8. GPU-accelerated building:
   - Vulkan compute pipeline (`lib/gpu/`)
   - Batched point insertion (`GPU_GROUPS_COUNT` = 512 parallel warps)
   - Half-precision option, 60-second timeout
   - Device management and fallback
9. `PlainVectorIndex`: brute force for small segments (below `SINGLE_THREADED_HNSW_BUILD_THRESHOLD` = 256)

## Part 7: Quantization & Scoring

**Goal:** How vectors are compressed and how distance is computed efficiently.

### TOC
1. Why quantize? (memory footprint, cache efficiency, faster distance ops)
2. Scalar quantization (U8): per-dimension min/max → 8-bit, 4x compression
3. Product quantization (PQ): partition dimensions → k-means codebooks → byte codes
4. Binary quantization: 1-bit per dimension, 32x compression, hamming distance
5. The `EncodedVectors` trait: common interface across methods
6. `VectorParameters`: dimension, distance type (Dot, L1, L2), invert flag
7. Rescoring pipeline: quantized rough pass → full-precision rescore top-k
8. Query scoring stack: `RawScorer` → `MetricQueryScorer` → `CustomQueryScorer`
9. Quantized storage variants: RAM, mmap, chunked mmap, multivector

## Part 10: The Distributed Life

**Goal:** Bootstrap, consensus, replication, shard transfer, replica state machine. Four sub-lifecycles in one post.

### TOC
1. **Bootstrap lifecycle:**
   - Node starts: parse args, load config, create 3 Tokio runtimes
   - TableOfContent loads collections from disk
   - WAL replay: apply pending operations from last checkpoint
   - Consensus: join cluster (bootstrap peer) or become leader (new cluster)
2. **Consensus (Raft):**
   - The consensus thread: `Consensus::run()`
   - `Message` enum: `FromClient` vs `FromPeer`
   - Proposal → log replication → commit → apply to ToC
   - Persistent state in RocksDB
   - gRPC transport between peers
3. **Replica state machine** (9 states):
   - `Active`, `Dead`, `Partial`, `Initializing`, `Listener`, `PartialSnapshot` (deprecated), `Recovery`, `Resharding`, `ReshardingScaleDown`
   - Valid transitions: Initializing → Active ↔ Listener, failure → Dead, transfer → Partial
   - Which states serve reads? Which accept writes?
4. **Shard transfer mechanisms:**
   - `ShardTransferMethod::Snapshot` — full snapshot
   - `ShardTransferMethod::StreamRecords` — record-by-record streaming
   - WAL delta — incremental catch-up
   - Resharding variant
   - Transfer driver: orchestration, task pool, error handling

## Cross-cutting Patterns to Highlight Across Posts

These patterns recur throughout the codebase and are worth calling out when they first appear:

1. **Mutable → Immutable → Mmap trilogy**: vector storage, payload index, field index, inverted index all follow this pattern
2. **Proxy/wrapper for concurrent modification**: ProxySegment wraps original during optimization
3. **Resource budgeting**: CPU/IO permits gate concurrent background work
4. **Trait-based polymorphism**: `SegmentEntry`, `VectorStorage`, `EncodedVectors`, `InvertedIndex`, `FieldIndexBase` — each subsystem has a trait + multiple implementations
